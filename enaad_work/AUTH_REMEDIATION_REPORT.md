# تقرير إصلاح المصادقة والأمان — مشروع لُغتي

## الحكم التنفيذي

تم تحديد السبب الجذري لفشل تسجيل الدخول وإصلاحه في المصدر، مع توحيد إعداد Better Auth وإضافة ترحيلات واختبارات تكامل حقيقية. نجحت اختبارات Better Auth عبر واجهاته الرسمية في إنشاء حساب وتخزين Hash وتسجيل الدخول وإنشاء Cookie وقراءة الجلسة وتدميرها وإعادة الدخول.

لا أعدّ النسخة مجازة نهائيًا للإنتاج قبل خطوتين خارج بيئة الفحص: تشغيل ترحيل Legacy على نسخة من قاعدة الإنتاج، وتشغيل Playwright بمتصفح حقيقي في CI/البيئة المرحلية. لم تكن بيانات قاعدة الإنتاج أو كلمات مرور مستخدمين متاحة، ولم تُطلب أو تُغيّر أي كلمة مرور.

## 1. السبب الجذري المحدد

كان `lib/auth.ts` الأصلي يفرض في وضع التطوير:

```ts
defaultCookieAttributes: { sameSite: 'none', secure: true }
```

على عنوان `http://localhost`. لذلك كان طلب `/api/auth/sign-in/email` ينجح ويعيد HTTP 200، ويستطيع Better Auth التحقق من الـHash وإنشاء الجلسة، لكن المتصفح يرفض حفظ Session Cookie ذات `Secure` على HTTP. عند الانتقال إلى الصفحة التالية لا توجد Cookie قابلة للقراءة، فيُعامل المستخدم كغير مسجل ويعود إلى صفحة الدخول.

هذا يفسر الحالة بدقة: صحة البريد وكلمة المرور لا تغيّر النتيجة لأن الفشل يحدث بعد التحقق، عند حفظ الجلسة في المتصفح.

وجد الفحص أيضًا مشكلتين مساعدتين:

- قائمة `trustedOrigins` الأصلية اعتمدت wildcards واسعة، وكان عنوان الخادم والعميل قابلًا للاختلاف.
- تفعيل Rate Limit المخزن في قاعدة البيانات يحتاج جدول Better Auth `rateLimit` بالأعمدة الحالية، ومنها `id`. المخطط الجديد والترحيل `0002` يثبتان الشكل المطلوب.

## 2. كيفية إعادة الإنتاج والتحقق

على النسخة الأصلية، أُرسلت طلبات التسجيل والدخول إلى Handler الرسمي لـBetter Auth باستخدام PostgreSQL dialect. ظهر ما يلي:

| المرحلة                    | النتيجة الأصلية                                                          |
| -------------------------- | ------------------------------------------------------------------------ |
| إنشاء حساب رسمي            | HTTP 200، مع سجل `user` وسجل `account` من نوع `credential` وHash غير نصي |
| دخول صحيح                  | HTTP 200                                                                 |
| `Set-Cookie` على localhost | `Secure` مع `SameSite=None` على HTTP؛ غير قابلة للحفظ في المتصفح         |
| الصفحة التالية             | لا تستطيع قراءة جلسة من Cookie متصفح صالحة                               |

بعد الإصلاح أثبت اختبار التكامل:

| الفحص                               | النتيجة                                                        |
| ----------------------------------- | -------------------------------------------------------------- |
| إنشاء حساب بالطريقة الرسمية         | HTTP 200                                                       |
| البريد بأحرف كبيرة ومسافات جانبية   | خُزن وقورن بصيغة موحدة                                         |
| كلمة المرور                         | Hash Scrypt المتوافق مع Better Auth، وليست نصًا عاديًا         |
| دخول صحيح                           | HTTP 200                                                       |
| Cookie localhost                    | `HttpOnly` و`SameSite=Lax` و`Path=/`، من دون `Secure` على HTTP |
| قراءة الجلسة مرتين/بعد انتقال منطقي | HTTP 200 ومستخدم حاضر في المرتين                               |
| خروج                                | الجلسة القديمة تصبح `null`                                     |
| إعادة الدخول                        | HTTP 200                                                       |
| كلمة مرور خاطئة وبريد غير موجود     | HTTP 401 ورسالة عربية عامة متطابقة                             |
| Origin غير موثوق                    | HTTP 403                                                       |
| كثرة المحاولات                      | HTTP 429                                                       |
| حساب موقوف                          | HTTP 403 برسالة مستقلة                                         |

لم يتوفر Chromium محليًا، وفشل تنزيل Playwright بسبب حجب/انقطاع ملف المتصفح في بيئة الفحص. لذلك لم أدّع تشغيل Network Panel أو الاختبار البصري محليًا. أُضيف اختبار E2E فعلي لسطح المكتب وPixel 7 إلى `tests/e2e/auth.spec.ts` وCI لتشغيله عند توفر المتصفح وPostgreSQL.

## 3. الإصلاحات المطبقة

### المصادقة والجلسة

- مصدر إعداد واحد في `lib/auth-config.ts` يستخدمه `lib/auth.ts` ومسار API والاختبارات.
- `basePath` ثابت: `/api/auth`، والعميل يستخدم Same-Origin بدل عنوان localhost مكتوب.
- `BETTER_AUTH_SECRET` ثابت بطول 32+ وإيقاف مبكر عند غيابه.
- Origins دقيقة بلا wildcard، مع فصل localhost عن عنوان الإنتاج.
- Cookie آمنة حسب البروتوكول: `Secure` على HTTPS فقط، وHost-only و`HttpOnly` و`SameSite=Lax`.
- تطبيع البريد قبل التسجيل والدخول وإرسال التأكيد؛ لا تُقص أو تُعدل كلمة المرور.
- Rate Limit في قاعدة البيانات للدخول والتسجيل وتأكيد البريد.
- رسائل عربية تفرق بين بيانات خاطئة، إيقاف الحساب، عدم تأكيد البريد، Rate Limit، فشل الجلسة، وخطأ الخادم؛ ولا تكشف وجود الحساب.
- سجلات المصادقة تحجب password/token/cookie/authorization ولا تطبع الأسرار.
- `proxy.ts` بوابة مركزية سريعة، بينما كل صفحة وServer Action تعيد التحقق من الجلسة والصلاحية على الخادم.
- مسار `/auth/continue` يقبل مسارًا داخليًا آمنًا فقط، يمنع Open Redirect ومسارات المصادقة المتكررة وحلقات التحويل.

### الحسابات والأدوار والاسترداد

- استبدال بريد المدير المكتوب بحقل `role` وقائمة أدوار وصلاحيات مخزنة في PostgreSQL.
- فصل نجاح المصادقة عن دخول الإدارة، مع حظر المستخدم العادي وتدقيق تغييرات الدور والحظر.
- أداة ترقية مدير لا تعمل إلا لحساب موجود ومؤكد، أو بتصريح مشغل صريح ومُدقق بعد تحقق خارجي من ملكية البريد.
- مسار تأكيد البريد الرسمي لـBetter Auth عند تهيئة Resend، مع زر إعادة الإرسال من صفحة الحساب.
- تعطيل مساري إعادة كلمة المرور المتداخلين داخل Better Auth واستخدام تدفق التطبيق الواحد.
- استهلاك رمز الاسترداد وتغيير الـHash وإبطال الجلسات وكل الرموز القديمة داخل Transaction واحدة.
- رابط تأكيد البريد الاحتياطي يعرض صفحة أولًا ولا يغير البيانات إلا عبر POST صريح.
- رموز عشوائية بطول 256-bit، مخزنة كـSHA-256 فقط، صالحة 30 دقيقة ومرة واحدة.
- Rate Limit مزدوج مبني على Hash للعنوان والمعرف، ورسالة طلب استرداد عامة تمنع Account Enumeration.

### نزاهة النتائج والمكافآت

- اختبار الكلمات يبدأ بمحاولة وأسئلة صادرة من الخادم، ويحفظ الإجابة الصحيحة في قاعدة البيانات ولا يقبل `isCorrect` من العميل.
- يرفض أي `wordId` أو سؤال لم يصدر للمحاولة، والتكرار والعدد الناقص والـPayload الأكبر من الحد.
- اختبارات الرياضيات والمنافسات تحفظ مجموعة الأسئلة الفعلية لكل محاولة/مشارك وتصحح على الخادم.
- كلمات المنافسة مقيدة بـ`userId` الحالي؛ لا يمكن جلب كلمة مستخدم آخر.
- نافذة المنافسة وحالتها والانضمام وحالة التسليم مطبقة داخل مسار التسليم نفسه.
- انتقال الحالة إلى `grading` ذري، وكل العمليات داخل Transactions.
- `rewardLedger` بقيد فريد `(userId, sourceType, sourceId)` يجعل XP وCoins Idempotent عند الطلبات المتزامنة.
- التحدي المكتمل لا يعود إلى نشط، وتواريخ البداية والنهاية مطبقة، و`rewardedAt` محفوظ.
- الشهادة فريدة لكل `(userId, challengeId)`، لها UUID عام وإلغاء، ورقم عشوائي 96-bit، وصفحة تحقق عامة منفصلة.

### قاعدة البيانات والملفات والتشغيل

- مخطط Drizzle شامل بعلاقات Foreign Keys وسياسات حذف وEnums وChecks وUnique Constraints وفهارس.
- Pool PostgreSQL واحد مشترك بين التطبيق وBetter Auth.
- أقسام منصة منفصلة عن تصنيفات المستخدم، وربط الكلمات بـ`categoryId`، وإدارة فعلية للأقسام من لوحة التحكم.
- حماية مسار الملفات بالملكية/الدور وسجل قاعدة البيانات وCanonicalization وقائمة MIME وContent-Disposition وRate Limit وPrivate Cache.
- فحص Magic Bytes وحجم صور البانرات، منع SVG، رابط زر HTTP(S)/داخلي فقط، وحذف Blob اليتيم عند فشل قاعدة البيانات.
- نقل قراءة XLSX من المتصفح إلى الخادم، وحد ملفين و2MB و2000 صف، وفحص بنية ZIP ونسبة الضغط والمسارات. أزيلت مكتبة `xlsx` ذات الثغرات.
- منع CSV Formula Injection، ودعم طباعة PDF العربية بخطوط النظام واتجاه RTL.
- CSP وHSTS وReferrer/Permissions Policy ومنع framing وMIME sniffing.
- ESLint وTypecheck وPrettier وCI وHealth Check وSeed وREADME و`.env.example`.

## 8. أثر المشكلة على الحسابات

- عطل Cookie على `http://localhost` يؤثر في الحسابات القديمة والجديدة معًا؛ التحقق من كلمة المرور قد ينجح لكن المتصفح لا يحتفظ بالجلسة.
- الحسابات الجديدة بعد الإصلاح تُنشأ ببريد موحد و`credential` Hash رسمي، وقد نجح اختبارها من أول محاولة.
- Hashes التي أنشأها إصدار Better Auth نفسه تظل متوافقة؛ لم تُستبدل أو تُعاد كلمات مرور المستخدمين.
- الترحيل يطبع البريد القديم بعد preflight، ويرفض التكرارات بدل حذفها.
- لا يمكن إثبات كلمة المرور الأصلية لحساب إنتاجي من Hash، وهذا مقصود أمنيًا. أداة `auth:diagnose` تفحص بنية الحسابات القديمة: غياب credential، Hash فارغ/غير متوافق، Provider-only، تكرار أو مسافات/حالة بريد، وجلسات/حسابات يتيمة، من دون طباعة Hash أو Token.
- لم تتوفر قاعدة الإنتاج في الملف المرفق؛ لذلك لم يُختبر دخول حساب حالي بكلمة مروره، ولم يُحذف حساب أو يُعد تعيين كلمة مرور. تدفق «نسيت كلمة المرور» الآمن متاح للحالات التي لا يمكن فيها معرفة كلمة المرور الأصلية.

## 9. حدود الإصدار المتبقية

أُغلقت ثغرات المصادقة والنتائج والمكافآت والعزل والملفات ذات الأولوية، وأضيفت إدارة الأقسام والمستخدمين الأساسية. لكن التقرير الأصلي يتضمن برنامج إعادة بناء أكبر من إصلاح المصادقة، وما زالت النقاط التشغيلية التالية تحتاج قرارًا قبل إعلان المنصة مكتملة إنتاجيًا:

- تشغيل Playwright فعليًا على بيئة مرحلية ومتصفح هاتف/سطح مكتب.
- تجربة Legacy migration على نسخة من بيانات الإنتاج ومعالجة أي تعارضات يبلغ عنها preflight يدويًا.
- تحويل «تصدير البيانات» اليدوي إلى منظومة نسخ مشفرة متدفقة تشمل PostgreSQL وBlob، مع Restore مجرب وPITR؛ الميزة الحالية ليست Disaster Recovery.
- استكمال Cursor Pagination وصفحات CRUD المتخصصة لكل جزء من لوحة الإدارة عند تجاوز الحدود الحالية.
- إضافة مراقبة خارجية وتنبيهات وTracing ومراجعة دورية للسجلات ومعدلات الاسترداد والمكافآت.

هذه الحدود معلنة حتى لا تُقدَّم نتيجة الاختبارات الآلية على أنها دليل تشغيل لمتصفح أو قاعدة إنتاج لم تكن متاحة.
