# سياسة الأمان

## الإبلاغ عن ثغرة

لا تنشر مفاتيح أو بيانات مستخدمين أو خطوات استغلال في Issue عامة. أرسل وصفًا خاصًا لمالك المشروع يتضمن المسار المتأثر، خطوات إعادة الإنتاج، الأثر، وبيئة الاختبار. لا تختبر على بيانات إنتاج حقيقية دون تفويض.

## الحدود الأمنية الرئيسية

- كل صفحة وإجراء إداري يتحقق من الجلسة والدور وحالة الحساب على الخادم.
- `INITIAL_ADMIN_EMAIL` مخصص لتهيئة أول مدير مرة واحدة فقط. تُستهلك التهيئة بعلامة دائمة في قاعدة البيانات، ولا تُستدعى أثناء قراءة الجلسة أو فتح الصفحات. بعد ذلك تكون قاعدة البيانات مصدر الحقيقة الوحيد للدور، ولا يعيد `seed` ترقية حساب خُفّضت صلاحيته. لا توجد دالة تفويض مستمرة تعتمد على مطابقة البريد.
- تغييرات دور المدير وحالة الحساب تستخدم قفلًا ومعاملة، وتعيد فحص صلاحية المنفذ داخل المعاملة، وتمنع تعطيل أو تخفيض آخر مدير نشط.
- العمليات الحساسة تتحقق من الملكية والمدخلات وتستخدم معاملات وقيودًا فريدة عند احتمال التكرار.
- نتائج الاختبارات والمنافسات والمكافآت تُحسب على الخادم ولا تُقبل من العميل.
- الملفات الخاصة تُخدم بعد التحقق من الملكية أو دور المدير، ولا تُكشف روابط Blob الخام.
- الوسائط العامة لا تُعرض إلا عند وجود مرجع منشور ونشط.
- صفحات الإدارة غير قابلة للفهرسة، لكن الحماية الفعلية تعتمد على الخادم لا على `robots`.
- رسائل الأخطاء العامة تُنقّى حتى لا تعرض SQL أو stack أو مفاتيح أو تفاصيل مزود الخدمة.

## الأسرار

لا تُحفظ القيم التالية في Git أو ملفات ZIP المسلّمة:

- `DATABASE_URL`
- `BETTER_AUTH_SECRET`
- `RESEND_API_KEY`
- `BLOB_READ_WRITE_TOKEN`
- `BACKUP_ENCRYPTION_SECRET`
- أي `.env` حقيقي

عند تسريب سر، غيّره لدى المزود ثم حدّث Vercel وأعد نشر التطبيق. تغيير `BETTER_AUTH_SECRET` يبطل الجلسات وقد يؤثر في الروابط الموقعة.

## البريد

- استخدم نطاق Resend موثّقًا للإنتاج.
- `onboarding@resend.dev` للاختبار المقيد فقط.
- لا تستخدم Gmail كعنوان مرسل ما لم يكن عبر مزود يدعمه ويوثقه صراحة.
- رسائل الاسترداد عامة لتقليل كشف وجود الحسابات.

## النسخ التشغيلية

- إنشاء النسخة متاح لصلاحية `backups:write` فقط.
- توجد قيود معدل ومنع نسخة متزامنة للمدير نفسه.
- كل ملف داخل الأرشيف له SHA-256، ويُعاد فتح ZIP والتحقق من CRC والبصمات وأعداد الجداول وملفات Blob قبل إعلان النجاح.
- تجزئات كلمات المرور اللازمة للاستعادة مشفرة بـAES-256-GCM داخل ملف مستقل.
- لا تتضمن النسخة sessions أو verification/recovery tokens أو مفاتيح البيئة أو OAuth tokens.
- رابط النسخة الكبيرة خاص وموقع ومحدد الصلاحية.
- الاستعادة ترفض قاعدة غير فارغة وتحتاج interlock صريحًا. استعادة قاعدة البيانات تتم داخل معاملة، لكن PostgreSQL وVercel Blob لا يقدمان معاملة موزعة مشتركة؛ لذلك يجب اعتبار استعادة الملفات خطوة خارجية قابلة للفشل ومراجعة نتيجتها قبل اعتماد البيئة.
- الحد الوقائي الحالي لمنشئ النسخة هو 60 MB لمجموع البيانات وملفات Blob؛ يتوقف الطلب بخطأ واضح عند تجاوزه بدل استهلاك ذاكرة غير منضبط أو إنشاء ZIP ناقص.

## رفع الملفات

- التحقق يعتمد على magic bytes وليس الامتداد أو MIME القادم من العميل وحده.
- توجد قائمة سماح للأنواع وحد أقصى للحجم وأسماء عشوائية.
- الاستبدال يحافظ على المراجع ويحذف الملف القديم بعد نجاح معاملة قاعدة البيانات.
- الاستجابات تستخدم `nosniff` وCSP وCache-Control مناسبًا.

## تشغيل آمن

قبل كل إصدار:

```bash
pnpm install --frozen-lockfile
pnpm db:migrate
pnpm lint
pnpm audit:schema
pnpm audit:a11y
pnpm typecheck
pnpm test
pnpm test:e2e
pnpm build
```

شغّل المهاجرات والاختبارات على قاعدة معزولة أولًا، واحتفظ بخطة رجوع ونسخة من مزود PostgreSQL.
